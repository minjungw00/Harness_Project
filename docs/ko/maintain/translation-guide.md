# 번역 가이드

영어와 한국어 하네스 문서를 함께 고칠 때 이 가이드를 사용합니다. 이 문서는 한영 의미 일치, 자연스러운 한국어 기술 문체, 식별자 보존 지침, 한국어 혼합어 문체, 숨김 앵커 방식을 담당합니다.

이 문서는 문서 유지보수 가이드입니다. 제품 동작, API 동작, 저장 효과, 보안 보장, 상태 보기 동작, 런타임 동작, 스키마 계약, 용어집 항목, 담당 문서 경로 색인을 정의하지 않습니다. 계약 세부사항이 필요하면 이 문서에 다시 쓰지 말고 집중 담당 문서로 연결합니다.

완전한 구조화 용어 메타데이터는 [`docs/terminology-map.yaml`](../../terminology-map.yaml)에 있습니다. [용어집](../reference/glossary.md)은 선별된 용어를 독자에게 간결하게 설명합니다. 제품 용어, 한국어 산문 용어, 식별자 설명, 제품 라벨, 한국어 혼합어 통제를 추가하거나 바꾸기 전에 용어 지도를 확인합니다.

## 1. 의미 일치

영어와 한국어 문서는 모두 유지되는 문서입니다. 어느 한쪽도 보관본, 부록, 번역 전용 사본이 아닙니다.

일치 기준은 줄 수가 아니라 의미 단위입니다. 의미 단위는 규칙, 주의 문장, 문단, 표의 행, 예시, 경로, 체크리스트 항목일 수 있습니다. 한국어가 더 분명해진다면 문장 순서를 바꾸거나, 문장을 나누거나 합치거나, 문단 흐름을 영어와 다르게 잡을 수 있습니다.

의미 일치는 같은 독자 목적, 규범 강도, 기준 범위와 지원 범위 밖 경계, 담당 문서 경로, 사용자 판단 경계, 증거 경계, 검증 경계, 수락 경계, 잔여 위험 경계, 보안 보장 수준, 정확한 식별자, 이름을 직접 말하는 정확한 제품 라벨을 유지하는 것입니다.

의미가 바뀌면 한 언어만 고친 채 끝내지 않습니다. 한국어를 다듬다가 영어 의미 문제가 보이면 영어도 함께 고칩니다. 영어에 제품 개념을 추가했다면 같은 문서 변경 범위에서 자연스러운 한국어 대응 표현을 한국어 문서에 넣습니다.

문장을 보기 좋게 다듬으려고 한국어에만 구조를 추가하지 않습니다. `조건`, `결과`, `비주장`, `허용되지 않는 것` 같은 라벨은 영어 문서에도 같은 의미 단위가 있을 때만 한국어에 둡니다.

## 2. 경로와 담당 문서

주요 유지 문서는 `docs/en/`과 `docs/ko/` 아래에 대응 경로를 둡니다. 대응 문서의 줄 번호는 같을 필요가 없지만, 범위와 독자 의도는 같아야 합니다.

경로와 담당 문서 참조는 의미를 보존합니다. 이 가이드에 전체 담당 문서 지도, API 메서드 지도, 저장 효과 요약, 스키마 필드 표, 보안 규칙, 제품 계약 문구를 다시 만들지 않습니다. 담당 문서 찾기는 [작성 가이드](authoring-guide.md), [`docs/doc-index.yaml`](../../doc-index.yaml), 적용되는 참조 담당 문서가 담당합니다.

일반 조회에서는 요청 언어 또는 `doc-index.yaml`의 기본 언어를 읽습니다. 번역, 한영 의미 일치 검토, 용어에 영향을 주는 편집에서는 두 언어를 함께 읽습니다.

## 3. 용어 결정

완전한 구조화 용어 기준은 [`docs/terminology-map.yaml`](../../terminology-map.yaml)입니다. 이 절은 짧은 문체 보조 자료일 뿐입니다.

아래 결정을 일관되게 적용합니다.

- 하네스는 AI 지원 제품 작업을 위한 로컬 작업 권한 제품이자 시스템입니다. Core는 하네스 상태를 위한 로컬 기준 기록입니다.
- 사용자가 작업을 확인할 때 쓰는 기준은 영어에서 "verification criteria", 한국어에서 "검증 기준"으로 씁니다.
- 산문 맥락에서는 영어 "current scope" 또는 "currently applied scope", 한국어 "현재 적용 범위"를 씁니다. `active`가 들어간 정확한 식별자와 상태 값은 그대로 보존합니다.
- 정확한 라벨 `Write Authorization`은 일반 쓰기 승인과 구분합니다. 한국어 설명에서는 `Write Authorization`을 "쓰기 권한 부여"로, 일반 쓰기 승인을 "쓰기 승인"으로 씁니다.
- 한국어 참조 문서에서 닫기 준비 상태 개념은 "닫기 준비 상태"로 씁니다.

용어 지도의 선호 표현 목록이나 피해야 할 표현 목록을 이 가이드나 용어집에 다시 복사하지 않습니다. 오래 유지될 용어, 제품 라벨, 식별자 설명, 한국어 혼합어 통제가 바뀌면 용어 지도를 추가하거나 갱신합니다.

## 4. 식별자와 라벨

정확한 식별자는 영어와 한국어에서 그대로 보존합니다. 코드, 스키마, 경로, 검색 기준으로 쓰이는 값은 산문에 나올 때 백틱으로 감쌉니다.

코드 블록, 스키마 예시, API 예시, 파일 경로, 필드 목록, 리터럴 값 표, 기계 판독용 메타데이터 안의 정확한 문자열은 번역하지 않습니다. 한국어 표시 라벨은 독자를 위한 문구일 뿐이며 기준 식별자나 정확한 제품 라벨을 대신하지 않습니다.

구분 예시는 아래와 같습니다.

- 정확한 식별자: `ArtifactRef`
- 식별자의 한국어 설명: 아티팩트 참조 스키마
- 일반 산문 개념: 아티팩트
- 정확한 제품 라벨: 라벨 자체를 말할 때의 `Product Repository`, `Harness Runtime Home`, `Projection`, `Write Authorization`
- 한국어 독자용 산문: 정확한 라벨 자체가 주제가 아닐 때의 제품 저장소, 런타임 홈, 상태 보기, 쓰기 권한 부여

영어 단어가 코드 값과 일반 형용사로 모두 쓰일 수 있으면 먼저 맥락을 판단합니다. `complete`는 `intent=complete`처럼 식별자일 때만 백틱으로 보존합니다. 일반 산문에서 full 또는 entire를 뜻한다면 영어는 "full"이나 "entire"를 선호하고, 한국어는 용어 지도의 일반 산문 치환을 따릅니다.

## 5. 한국어 산문

한국어 문서는 영어 문장을 비춘 번역문이 아니라 한국어 기술 문서로 읽혀야 합니다.

한국어는 자연스러운 제목, 짧고 설명적인 문장, 한국어 개념을 먼저 두는 흐름, 용어 지도의 일관된 표현, 한국어 독자가 이해할 만큼의 맥락, 필요한 곳의 백틱 식별자를 갖추어야 합니다.

일반 영어 명사와 명사구는 한국어 산문으로 옮깁니다. 영어를 그대로 둘 수 있는 경우는 정확한 식별자, 파일 경로, 앵커, 코드 리터럴, 스키마 값, enum 값, 테이블이나 필드 이름, API 메서드, 의도적 하네스 제품 라벨, API, SDK, MCP, YAML, JSON, QA, CLI처럼 한국어 기술 문서에서 자연스러운 차용어입니다.

식별자나 제품 라벨이 아닌 영어 명사구에 한국어 조사만 붙이는 표현을 피합니다. 먼저 한국어 개념으로 설명하고, 계약상 정확성이나 검색성이 필요할 때만 정확한 영어 값을 덧붙입니다.

유지보수 문구와 문서 점검 문구는 편집 맥락이 사라진 뒤에도 안정적으로 읽혀야 합니다. 일반 유지보수 규칙에는 `변경된`, `편집된`, `문서 변경 시`, `점검 대상` 같은 표현을 사용합니다.

## 6. 예시의 한영 처리

예시가 대응 문서에 함께 나타날 때는 같은 시나리오 의미를 보존하되 한국어는 자연스럽게 씁니다. 시나리오가 소개된 뒤에는 의미가 유지되는 범위에서 반복 표현을 줄일 수 있지만, 관련 예시 사이에서는 의도적으로 다른 시나리오가 아닌 한 표현을 일관되게 유지합니다.

영어 원문이 짧은 명사구를 사용했다는 이유만으로 한국어 예시에서 영어 어순을 보존하지 않습니다. 참조, 경로, 메서드 이름, 스키마 필드, 상태 값, 제품 라벨은 그대로 둡니다.

예시 품질 규칙, 오래 유지되는 시나리오 점검, 내부 정합성 점검에는 [작성 가이드](authoring-guide.md)와 [API 예시 점검](checks/api-examples.md)을 사용합니다.

## 7. 숨김 앵커

기존 링크나 외부 북마크 때문에 안정적인 영어 앵커가 필요할 수 있습니다. 이때는 자연스러운 한국어 제목 앞에 숨김 HTML 앵커를 둡니다.

형식은 아래와 같습니다.

```markdown
<a id="close-readiness"></a>
## 닫기 준비 상태
```

영어 앵커에 맞추려고 보이는 한국어 제목을 어색하게 만들지 않습니다. 앵커는 링크 안정성을 담당하고, 제목은 독자를 담당합니다.

링크 텍스트는 보이는 제목의 의미와 맞아야 합니다. 보이는 제목이 `## 닫기 준비 상태`라면 링크 텍스트도 "닫기 준비 상태"처럼 씁니다.

한 언어에서 제목을 바꾸면 같은 문서 변경 범위에서 대응 언어의 링크와 앵커도 확인합니다.

## 8. 검토

번역을 고친 뒤에는 이 가이드를 체크리스트로 쓰지 말고 집중 유지보수 점검을 실행합니다.

- [언어 일치 점검](checks/language-parity.md): 의미 단위 일치, 자연스러운 한국어 구조, 제목, 표, 목록, 식별자 보존을 확인합니다.
- [용어 점검](checks/terminology.md): 용어 지도 정렬, 혼합어 한국어, 문서 경로 안내 용어, 현재 맥락 표현, `complete` 모호성, 닫기 준비 상태 표현, 관련 용어 통제를 확인합니다.
- [링크와 색인 점검](checks/links-and-indexes.md): 제목, 앵커, 상대 링크, 용어 담당 대상, 경로 메타데이터가 바뀌었을 때 사용합니다.
