# 그림 정책

Volicord 문서에서 그림을 만들고, 검토하고, 캡션을 붙이고, 유지할 때 이 정책을
사용합니다. 이 문서는 [문서 정책](documentation-policy.md),
[문서 헌장](document-charters.md), [브랜드 지침](brand-guidelines.md),
[검증](validation.md)을 보완합니다.

이 문서는 유지보수 정책입니다. 제품 동작, API 동작, 저장 효과, 보안 보장,
런타임 동작, 스키마, Core 권한 의미, 적합성 결과, QA 결과, 수락 판단, 닫기
준비 상태, 잔여 위험 판단을 정의하지 않습니다.

## 목적

모든 그림은 하나의 독자 작업에 답해야 합니다. 화살표를 그리기 전에 그림 범주를
고르고, 그 독자 작업이 주변 산문이나 캡션에서 분명하게 드러나게 합니다. 그림
범주와 역할 메타데이터는 작성과 검토를 돕는 자료입니다. 유지보수자가 그림의
위치, 캡션 범위, 검증 기대를 판단하는 데 쓰는 것이며, 독자에게 반드시 보이는
라벨이 아닙니다. 하나의 그림이 작업 흐름 순서, 구성 요소 책임, 권한, 런타임 호출,
저장 이동을 동시에 보여 주어야 한다면 더 작은 그림으로 나누거나 집중 담당 문서로
세부사항을 옮깁니다.

그림은 설명을 돕는 자료이지 계약을 대체하지 않습니다. 정확한 동작, 저장 효과,
스키마 의미, 보안 표현, API 동작, Core 권한 의미가 중요하다면 그림은 적용되는
참조 담당 문서나 적용되는 담당 문서로 연결해야 합니다. 참조 문서처럼 보이는 그림에는
분명한 정확성 담당자가 있어야 하며, 그렇지 않으면 추가하지 않습니다.

## 캡션에 필요한 맥락

간단한 그림은 한 줄의 간결한 캡션으로 이 요구를 만족할 수 있습니다. 복잡한
Mermaid 그림, 참조 문서 안의 그림, 여러 화살표 스타일을 쓰는 그림은 짧은 캡션과
가까운 설명 또는 범례를 함께 사용해야 합니다.

캡션은 그림이 무엇을 보여 주는지, 필요할 때 무엇을 생략하는지 설명해야 합니다.
관련될 때 캡션이나 가까운 산문은 아래를 분명히 해야 합니다.

- 그림이 답하는 질문
- 대상 독자
- 각 화살표 스타일의 의미
- 그림을 정확하게 유지하는 담당자 또는 기준 정보 출처
- 그림이 의도적으로 생략하는 것

그림 제목에만 기대지 않습니다. 독자가 화살표 의미를 이해하려고 원문 소스를
살펴봐야 한다면 캡션이 충분하지 않습니다.

## 독자용 캡션과 역할 메타데이터

역할 메타데이터는 작성 메모, 검토 의견, 유지보수 정책 논의에서 검토자가 왜 그
그림이 해당 문서에 속하는지 이해하는 데 도움이 될 때 사용합니다. 일반 독자용
산문에 그대로 붙여 넣지 않습니다.

README, 시작하기, 가이드, 참조, 개발 문서에서는 `Diagram role:`이나 `그림 역할:`
같은 문자 그대로의 정책 라벨을 피합니다. 독자용 문서는 같은 목적을 자연스럽게
설명해야 합니다. 예시는 아래와 같습니다.

- "이 작업 흐름은 사용자, 에이전트 호스트, Volicord가 결정을 어떻게 인계하는지
  보여 주며 저장소 배치와 API 호출 순서는 생략합니다."
- "이 지도는 호스트가 시작하거나 읽는 로컬 구성 요소를 보여 주며 완전한 런타임
  순서가 아닙니다."

"작업 흐름"이나 "구성 요소 지도"가 가장 분명한 독자용 설명이라면 사용할 수
있습니다. 일반 독자가 그림을 이해하기 전에 유지보수 메타데이터를 해석해야 하게
만들지 않습니다.

## 그림 범주

| 범주 | 답하는 질문 | 일반적인 독자 | 화살표가 보통 뜻하는 것 | 보여 주는 데 쓰면 안 되는 것 | 적절한 위치 |
|---|---|---|---|---|---|
| 사용자 작업 흐름 그림 (`User workflow diagram`) | 사용자나 에이전트가 다음에 무엇을 하며, 판단이나 인계가 어디서 일어나는가? | 새 사용자, 제품 사용자, 에이전트, 운영자 | 작업 순서, 인계, 사용자 결정, 또는 보이는 협업 루프 | 아키텍처 구성 요소, API 호출 순서, 저장소 소유, 권한 기준 정보 출처 | 루트 README, 시작하기 개요, 사용자 작업 흐름 가이드, 에이전트 작업 흐름 가이드, 가이드 수준 예시 |
| 구성 요소 지도 (`Component map`) | 어떤 부분들이 있으며, 각 부분은 어떤 책임이나 경계를 갖는가? | 운영자, 소스 코드 학습자, 구현자 | 범례가 정의한 관계, 포함, 책임 경계, 또는 허용된 통신 | 사용자 작업 흐름, 정확한 런타임 순서, 사용자 판단의 소유, 저장 생명주기 | 개요 깊이의 루트 README, 에이전트 호스트 설정, 아키텍처와 개발 문서, 계약으로 뒷받침되는 집중 참조 담당 문서 |
| 런타임 순서 (`Runtime sequence`) | 호출, 메시지, 프로세스 단계가 어떤 시간 순서로 일어나는가? | 구현자, 에이전트 통합 담당자, 검토자 | 시간 순서가 있는 호출, 메시지, 응답, 콜백, 또는 프로세스 전이 | 권한 의존성, 사용자 승인 의미, 오래 유지되는 저장소 소유, 넓은 구성 요소 책임 | 개발 문서, 요청 생명주기 설명, API 메서드 담당 문서, 집중 참조 문서 |
| 권한 모델 (`Authority model`) | 어떤 기록, 역할, 채널이 결정, 기준 정보 출처, 권한 경계를 담당하는가? | 유지보수자, 에이전트, 검토자, 구현자 | 권한 관계, 소유, 기준 정보 출처 경로, 또는 결정 경로 지정 관계 | 실행 순서, 구현 호출 스택, 저장 위치, 구성 요소 배포 | Core 모델, 사용자 작업 흐름 가이드, 에이전트 작업 흐름 가이드, 참조 담당 문서, 개념 깊이에 머무르는 개요 문서 |
| 저장 생명주기 (`Storage lifecycle`) | 저장된 기록, 아티팩트, 파일 상태가 생성, 갱신, 보존, 제거를 거치며 어떻게 이동하는가? | 구현자, 저장소 검토자, 유지보수자 | 상태 전이, 지속 경계, 보존 전이, 또는 삭제/제거 단계 | 프로세스 소유, 권한 소유, 사용자 작업 흐름 순서, 런타임 호출 순서 | 저장소 참조, 런타임 경계, 개발 아키텍처, 저장소 중심 구현 문서 |
| 연결 설정 흐름 (`Connection setup flow`) | 운영자가 Agent Connection 설정을 어떻게 준비, 검증, 안내, 사용, 제거하는가? | 운영자, 에이전트 통합 담당자, 에이전트 | 설정 단계, 검증 체크포인트, 설정 인계, 또는 복구 분기 | 전체 MCP 프로토콜 순서, 저장 생명주기, 넓은 아키텍처 지도, 정확한 CLI 출력 계약 | 빠른 시작의 간결한 성공 경로, 에이전트 호스트 설정, 에이전트 호스트 문제 해결, 명령 계약 깊이의 관리 CLI |
| 의존성 그래프 (`Dependency graph`) | 빌드, 소스 구성, 문서 담당 범위, 개념 경로에서 무엇이 무엇에 의존하는가? | 구현자, 유지보수자, 검토자, 소스 코드 학습자 | 캡션이 선언한 의존 방향. 예: 빌드 의존성, 모듈 의존성, 문서 의존성, 개념 전제 | 런타임 실행 순서, 권한 소유, 저장 이동, 사용자 작업 순서 | 개발 문서, 아키텍처 문서, 테스트 전략, Maintain 문서, 경로 메타데이터 설명 |

## 검토 규칙

- 그림 범주를 문서 헌장에 맞춥니다. 첫 온보딩 문서는 복잡한 구현 순서도를
  담으면 안 되고, 참조 문서는 넓은 안내용 그림을 정확한 동작의 출처로 사용하면
  안 됩니다.
- 검토 메타데이터와 독자용 산문을 분리합니다. 독자용 캡션은 유지보수 라벨을
  노출하지 않고 그림의 목적, 화살표 의미, 생략 사항을 설명해야 합니다.
- 한 화살표 스타일에는 하나의 의미만 둡니다. 실행 흐름과 권한 의존성이 함께
  나온다면 서로 다른 스타일과 범례를 사용하거나 그림을 나눕니다.
- 저장 위치, 프로세스 소유, 권한 소유를 구분합니다. 파일 경로, 프로세스, 구성
  요소가 둘 이상의 관계에 나타날 수는 있지만, 그림이 그 관계들을 같은 것처럼
  보이게 하면 안 됩니다.
- 구성 요소 지도는 독자가 부분과 경계를 이해하게 할 때 사용합니다. 아키텍처 구성
  요소 그림을 사용자 작업 흐름의 대체물로 사용하지 않습니다.
- 복잡한 Mermaid 그림은 의도한 독자가 세부사항을 기대하는 위치에 둡니다. 첫
  독자에게 제품 방향을 알려야 한다면 간결한 작업 흐름이나 개념 지도를 선호하고 더
  깊은 담당 문서로 연결합니다.
- 설명 캡션, 범례, 생략한 세부사항을 이름 붙이는 가까운 산문 없이 복잡한 Mermaid
  그림을 추가하지 않습니다.
- 제품 주장이나 시각 표현의 비중이 있는 그림은 [브랜드 지침](brand-guidelines.md)을
  따릅니다. 색만으로 상태를 전달하지 않습니다.

## 정확성 유지

그림이 속한 담당 문서의 의미가 바뀌거나, 그림이 요약하는 사실을 가진 참조 담당
문서가 바뀌거나, 개발 문서의 그림에 영향을 주는 오래 유지될 소스 구조가 바뀌면
그림을 검토합니다.

그림을 유지하거나 추가하기 전에 아래를 확인합니다.

- 범주와 독자 작업이 분명합니다.
- 필요할 때 캡션이 화살표 의미와 생략 사항을 이름 붙입니다.
- 그림이 그 수준의 세부사항을 담당할 수 있는 문서에 있습니다.
- 정확한 동작은 그림만으로 정의하지 않고 집중 참조 담당 문서로 연결합니다.
- 바뀐 문서가 한영 대응 문서라면 영어와 한국어 문서가 같은 의미를 보존합니다.

분명한 담당자와 검토 경로로 정확하게 유지할 수 없는 그림은 산문으로 바꾸거나 유지
문서에서 제거합니다.
