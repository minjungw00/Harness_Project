# 문서 점검

문서 편집 뒤와 큰 리뷰 인계 전에 이 문서 점검을 사용합니다. 이 점검은 Markdown 문서를 대상으로 하는 읽기 전용 점검입니다. 런타임 점검이 아닙니다.

`PASS`, `WARN`, `FAIL`은 문서 유지보수용 라벨로만 사용합니다. 문서 수락, 구현 준비, 런타임 적합성, 최종 수락, 닫기 준비 상태, 잔여 위험 수락, QA, 서버 코딩 시작 허가를 결정하지 않습니다.

## 1. 무엇을 점검하는가

이 문서는 문서 불일치를 찾습니다.

- 경로 문서가 담당 문서로 안내하지 않고 계약을 정의하는 문제
- 깨진 링크, 오래된 경로, 오래된 경로 묶음
- 이중 언어 의미 일치 문제
- 줄 단위 영어 번역처럼 읽히는 한국어 문장
- 정확한 식별자를 번역하거나 잘못된 문맥에서 표시 라벨처럼 쓰는 문제
- [참조 색인](../reference/README.md)이 지정한 기준 담당 문서 밖에서 반복되는 엄격한 계약
- active/later 경계 불일치와 profile-gated 값을 기본 활성 값처럼 보이는 문제
- 담당 문서가 문서화한 보장 수준보다 강한 보안 주장
- 서로를 대체하는 사용자 소유 판단 경로
- 렌더링된 상태 보기나 템플릿을 원천 권한처럼 다루는 문제
- 하나의 `doc_id`에 두 언어를 함께 싣는 검색 문제
- 오래된 재작성 이력, 종료된 이슈 기록, 마이그레이션 메모, 임시 파일

## 2. 무엇을 증명하지 않는가

이 문서는 런타임 동작, 런타임 적합성, 구현 준비 상태, 문서 수락, 개발 준비 상태, 최종 수락, 닫기 준비 상태, QA, 증거 충분성, 잔여 위험 수락, 서버 코딩 시작 허가를 증명하지 않습니다.

이 점검을 사용해 런타임 상태, `task_events`, 생성된 상태 보기, 생성된 운영 산출물, 실행 가능한 fixture, 적합성 보고서, QA 기록, 수락 기록, 닫기 기록, 잔여 위험 기록, 제품 쓰기를 만들지 않습니다.

## 3. 경로 점검

README, 유지보수 문서, 경로 표, 탐색 요약, 대응 언어 링크, 검색 지침을 봅니다.

경로 문서가 아래 항목으로 독자를 안내하면 통과입니다.

- `docs/doc-index.yaml`
- `docs/*/start.md`
- `docs/*/use/user-guide.md`
- `docs/*/use/agent-guide.md`
- `docs/*/use/judgment-examples.md`
- `docs/*/use/surface-recipes.md`
- `docs/*/build/mvp-plan.md`
- `docs/*/reference/README.md`
- `docs/*/later/index.md`
- `docs/*/maintain/authoring-guide.md`
- `docs/*/maintain/translation-guide.md`
- `docs/*/maintain/checks.md`

활성 경로가 삭제된 파일, 오래된 경로 묶음, 비활성 마이그레이션 기록, 잘못된 언어의 담당 문서, 또는 간결한 담당 색인을 써야 하는 자리에 깊은 담당 파일을 가리키면 실패입니다.

## 4. 담당 문서 경계 점검

[참조 색인](../reference/README.md)에서 계약 영역의 기준 담당 문서를 찾습니다. 정확한 스키마, DDL, enum 값, 상태 전이, 알고리즘, 템플릿 본문, 저장소 규칙, 보안 보장, 공개 오류, 공식 정의는 그 담당 문서에 있는지 확인하는 데 필요한 만큼만 봅니다.

엄격한 계약이 담당 문서 하나에서만 정의되고, 담당 문서가 아닌 문서가 짧은 결과 설명과 경로만 두면 통과입니다. README, Start, Use, Build, Maintain, 예시, 담당 문서가 아닌 참조 요약이 두 번째 규범 정의를 만들면 실패입니다.

## 5. active/later 점검

활성 문서, 예시, 경로 문구가 이후 전용 자료를 현재 MVP 범위처럼 보여 주는지 봅니다.

활성 자료가 활성 담당 문서에 있고, profile-gated 값이 사용하는 자리에서 표시되며, 이후 후보가 승격 전까지 [이후 후보 색인](../later/index.md)에 남으면 통과입니다. 이후 후보, 향후 운영, 프로필 전용 값, 증명되지 않은 보안 동작이 기본 활성 요구사항처럼 보이면 실패입니다.

## 6. 이중 언어 점검

영어와 한국어 대응 파일이 같은 활성 파일 지도, 독자 목적, 섹션 범위, 담당 문서 경로, active/later 경계, 정확한 식별자를 유지하는지 봅니다.

의미가 같고 한국어가 자연스러우면 통과입니다. 한 언어가 의미 변경을 빠뜨리거나, 담당 문서 경로를 바꾸거나, 정확한 식별자를 번역하거나, 이후 자료를 활성 범위로 만들면 실패입니다.

## 7. 한국어 품질 점검

한국어 설명 문장, 제목, 예시, 유지보수 지침을 봅니다.

자연스러운 한국어 기술 문서처럼 읽히고, 정확한 식별자를 보존하며, 정확한 식별자나 의도적인 하네스 라벨이 아닌 영어 명사구를 남기지 않으면 통과입니다. 자연스러운 한국어 설명이 필요한 자리에서 영어 문장 순서를 보존하거나, 부정 조건을 압축해 의미를 바꾸면 실패입니다.

## 8. 보안 표현 점검

`cooperative`, `detective`, `preventive`, `isolated`, guard, freeze, careful-mode, 샌드박스, 권한, 차단, 변조 방지, 격리, `surface_id`, 역량, 로컬 접근, 접점 바인딩 표현을 봅니다.

보장 의미가 보안 담당 문서로 연결되고 문서화된 메커니즘보다 강하게 말하지 않으면 통과입니다. 담당 문서가 뒷받침하지 않는데 OS 권한, 임의 도구 샌드박스, 변조 방지 저장소, 기본 도구 실행 전 차단, 보안 격리, 검증되지 않은 탐지 역량을 암시하면 실패입니다.

## 9. 판단과 증거 점검

판단 질문, 예시, 닫기 표현, 승인 표현, 최종 수락 표현, 잔여 위험 표현, 증거 표현을 봅니다.

제품 판단, 기술 판단, 범위 판단, 민감 동작 승인, 최종 수락, 잔여 위험 수락, 취소 판단, later/reserved 판단 경로가 구분되면 통과입니다. 넓은 승인이나 어느 한 판단이 다른 판단을 대신하거나, 수락 표현이 빠진 필수 증거를 채우는 것처럼 보이면 실패입니다.

## 10. 상태 보기와 템플릿 점검

상태 보기, 템플릿, 상태 카드, 요약, 생성된 표시 문구를 봅니다.

렌더링된 출력이 읽기 전용 파생 표시 또는 지원 맥락으로 설명되고 정확한 템플릿 본문이 [템플릿 본문](../reference/template-bodies.md)에 있으면 통과입니다. 렌더링된 출력이 원천 상태, 증거, QA, 수락, Write Authorization, 잔여 위험 수락, 닫기 결과, 제품/런타임 쓰기 허가처럼 다뤄지면 실패입니다.

## 11. 검색 점검

에이전트 지침, 맥락 적재 조언, README 경로, 참조 경로, 상시 맥락 예시를 봅니다.

일반 작업에서 하나의 `doc_id`에는 한 언어만 검색하고, 번역이나 의미 일치 검토 때만 대응 언어를 함께 읽으며, 다음 행동에 필요한 담당 섹션만 가져오고, 맥락을 작게 유지하면 통과입니다.

## 12. 오래된 내용 점검

유지보수 문서와 경로 문구에서 과거 재작성 리뷰, 종료된 이슈 기록, 오래된 수락 기록, 오래된 별칭 이력, 과거 지역화 점검, 과거 번역 문제 기록, 임시 마이그레이션 계획을 찾습니다.

활성 문서에 오래 남는 규칙과 담당 문서에 맞는 내용만 있으면 통과입니다. 오래된 이력이 active 지침처럼 남거나 임시 파일, 보관 사본, 마이그레이션 메모가 편집 뒤에도 남으면 실패입니다.
