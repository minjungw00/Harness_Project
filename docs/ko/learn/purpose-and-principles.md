# 목적과 원칙

## 여기서 시작하기

짧은 학습 경로를 읽은 뒤, 문서의 문구가 하네스 제품 명제와 맞는지 검토할 때 봅니다. 처음 읽는 사람은 [개요](overview.md)와 [하나의 작업](one-task.md)부터 보면 됩니다.

이 문서는 원칙 문서입니다. 구현 상태 보고서가 아닙니다. 이 저장소는 문서 전용이며 하네스 서버 런타임을 담고 있지 않습니다.

## 목적

하네스는 AI 지원 제품 작업을 따라갈 수 있게 만들면서 사용자 판단권을 보존하기 위해 존재합니다.

사용자는 평소 말로 시작할 수 있어야 합니다. 에이전트는 구체화하고, 살펴보고, 추천하고, 필요하면 구현하고, 확인하고, 보고할 수 있어야 합니다. 조심해야 할 부분은 권한입니다. 범위, 사용자 소유 결정, 근거, 검증, QA 기대, 작업 수락, 닫기 가능 여부, 잔여 위험이 대화 표현, 생성된 Markdown, 커넥터 출력, 테스트 로그, 에이전트 자신감으로 새어 나가면 안 됩니다.

## 원칙

하네스는 권한을 로컬에 둡니다. 오래 남는 작업 근거가 원격 대화 기록이나 생성된 보고서에 의존해서는 안 됩니다.

하네스는 서로 다른 것을 분리합니다. 범위, 민감 단계 허가, 제품 판단, 기술 판단, 근거, 검증, 수동 QA, 작업 수락, 잔여 위험 수용은 서로 다른 질문입니다.

하네스는 사용자 판단권을 보존합니다. 목표, 범위, 제품/UX 판단, 중요한 기술 판단, QA 기대, 작업 수락, 잔여 위험 수용은 사용자가 소유합니다.

하네스는 근거를 과장하지 않습니다. 무엇을 확인했는지, 어떤 종류의 확인인지, 무엇이 주장을 뒷받침하는지, 아직 무엇이 검증되지 않았는지, 어떤 부분에 사람의 확인이 필요한지 말해야 합니다.

하네스는 작은 작업을 작게 유지합니다. 좁은 오탈자, 문구 변경, 작은 수정에는 무거운 절차가 필요하지 않습니다. 하지만 범위, 의미, 위험, UX, 공개 동작, 민감 동작, 공유 계약 영향이 나타나면 더 이상 작은 작업처럼 다루면 안 됩니다.

하네스는 실제 능력으로 보장을 설명합니다. 협력형 안내는 사전 차단과 다릅니다. 초기 하네스 문구는 OS 수준 격리, 임의 도구 샌드박스, 변조 방지 저장을 암시하면 안 됩니다.

## 실패 모델

하네스는 AI 지원 작업에서 반복되는 실패를 기준으로 설계됩니다.

- 범위가 암묵적으로 바뀝니다.
- 사용자 판단이 구현 속에서 조용히 대체됩니다.
- 근거, 검증, QA, 수락, 위험이 하나의 "끝남"으로 합쳐집니다.
- 대화, 도구 출력, Markdown이 잘 쓰였다는 이유로 권한처럼 보입니다.

하네스는 경계를 보이게 하고, 서로 다른 지원을 각자의 자리에서 다루게 합니다.

## 비목표

하네스는 제품 저장소, 버전 관리, 테스트, 리뷰, 제품 사양, 사용자 판단, 팀 절차를 대체하지 않습니다.

하네스는 프롬프트 묶음, MCP 자체, API 래퍼, 워크플로 엔진, 보고서 생성기, 대시보드, 호스팅 에이전트 플랫폼, 샌드박스, OS 권한 시스템이 아닙니다.

하네스는 지침, 도구 출력, 테스트, 리뷰, 보고서, 대시보드, 사양을 맥락이나 근거 출처로 사용할 수 있습니다. 유용하다고 해서 하네스 권한이 되지는 않습니다.

## MVP 경계

첫 향후 구현 작업은 넓은 플랫폼을 만드는 것이 아니라 로컬 작업 권한 모델을 증명하는 데 있습니다.

초기 조각은 평범한 AI 지원 작업을 로컬 범위, 사용자 소유 판단, 근거 참조, 닫기 가능 여부, 작업 수락, 잔여 위험으로 표현할 수 있음을 보여줘야 합니다. 그 기록이 대화, Markdown, 도구 출력, 제품 파일과 섞이면 안 됩니다.

더 넓은 자동화, 더 풍부한 읽기용 보기, 커넥터 생태계, 호스팅 워크플로, 대시보드, 분석, 큰 conformance suite는 owner 문서가 나중에 승격하기 전까지 첫 사용자 가치 명제 밖에 둡니다.
