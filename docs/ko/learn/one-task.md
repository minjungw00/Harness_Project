# 하나의 작업으로 보는 하네스

## 평소 요청 하나에서 시작하기

이 문서는 향후 하네스가 붙은 작업 하나를 따라갑니다. 학습용 이야기이지 런타임 기록이 아닙니다. 이 저장소는 문서 전용이며 실행 중인 하네스 서버를 담고 있지 않습니다.

사용자가 이렇게 요청합니다.

```text
로그인에 "로그인 유지" 동작을 추가해줘. 구현 전에 계획부터 구체화해줘.
```

사용자는 내부 라벨을 알 필요가 없습니다. 에이전트는 쉬운 질문으로 작업을 보이게 해야 합니다. 무엇을 하려는가? 무엇이 범위인가? 사용자가 무엇을 결정해야 하는가? 어떤 근거가 결과를 뒷받침하는가? 무엇을 확인했는가? 무엇이 아직 닫기를 막는가?

## 1. 쓰기 전에 구체화하기

요청 안에 제품 선택이나 보안 선택이 숨어 있으면 에이전트는 파일부터 바꾸면 안 됩니다.

```text
내가 살펴볼 수 있는 것:
- 현재 로그인 폼 구조;
- 현재 세션 수명 설정;
- 로그인과 로그아웃 관련 기존 테스트.

사용자가 결정해야 할 가능성이 있는 것:
- "로그인 유지"가 로그인 상태 유지인지, 이메일 기억인지, 둘 다인지;
- 로그인 상태를 유지한다면 어떤 세션 수명과 저장 위험을 받아들일 수 있는지.

안전한 다음 단계:
인증 구조를 살피고 좁은 계획을 돌려준다. 아직 제품 파일은 바꾸지 않는다.
```

계획은 유용하지만 구현 허가도 아니고 위험 수용도 아닙니다.

## 2. 사실과 사용자 판단 나누기

살펴본 뒤 에이전트는 이렇게 요약할 수 있습니다.

```text
확인된 사실:
- 로그인 폼은 레이아웃 재작성 없이 체크박스를 추가할 수 있습니다.
- 세션 수명은 서버 쪽에서 제어됩니다.
- 테스트는 이미 로그인과 로그아웃을 다룹니다.

아직 사용자가 결정할 것:
- "로그인 유지"의 제품 의미;
- 세션을 연장한다면 받아들일 수 있는 세션 수명.
```

사용자가 이렇게 말하면:

```text
내가 결정해야 할 것과 네가 확인할 수 있는 것을 나눠서 보여줘.
```

에이전트는 그 구분을 계속 보여줘야 합니다.

```text
사용자 결정:
"로그인 유지"가 이메일 기억인지, 더 긴 세션인지, 둘 다인지.

에이전트가 확인할 수 있는 것:
기존 세션 방식이 선택한 동작을 지원할 수 있는지.
```

도구 출력은 선택에 도움을 줄 수 있습니다. 사용자의 선택을 대신하지는 않습니다.

## 3. 범위 이름 붙이기

사용자가 더 긴 세션을 선택하면, 에이전트는 구현 전에 작업 경계를 보여줘야 합니다.

```text
범위 안:
로그인 체크박스, 선택한 세션 연장 동작, 집중 테스트, 직접 관련된 문구.

범위 밖:
비밀번호 없는 로그인, 계정 복구, 전체 세션 재설계, 관련 없는 인증 정리.

필요하면 여전히 사용자 소유:
세션 수명, 저장 위험, QA 기대, 잔여 위험 수용, 최종 작업 수락.
```

중간에 "로그인 버튼 문구도 Sign in에서 Log in으로 바꿔줘" 같은 작은 요청이 나오면, 문구가 로컬이고 의미가 그대로인 동안만 가볍게 처리할 수 있습니다. 그 문구가 다른 흐름에서도 공유된다면 멈추고 범위가 커졌다고 말해야 합니다.

## 4. 구체적으로 묻기

하네스가 모든 불확실성을 무거운 절차로 만들 필요는 없습니다. 정직한 진행을 막는 질문을 구체적으로 물으면 됩니다.

```text
결정할 것:
이 기기에서 로그인 유지 세션을 얼마나 오래 유지할까요?

추천:
별도 지속 토큰을 바로 추가하기보다 기존 세션 방식을 짧은 연장 시간으로 쓰는 쪽을 먼저 검토합니다.

불확실성:
현재 세션 정리 로직이 연장된 수명을 어떻게 처리하는지 아직 확인해야 합니다.
```

막연한 "진행해"가 관련 없는 제품, 기술, QA, 수락, 위험 판단까지 모두 답한 것으로 늘어나면 안 됩니다.

## 5. 민감한 허가는 좁게 유지하기

선택한 접근에 민감한 단계가 필요하면, 질문은 무엇을 허가하는지 정확히 말해야 합니다.

```text
민감 단계 허가가 필요합니다:
이 접근에 필요한 세션 helper를 추가하기 위해 의존성 lockfile을 업데이트해도 될까요?

이 허가는 그 단계에만 적용됩니다. 완료 결과 수락, QA 면제, 잔여 위험 수용을 뜻하지 않습니다.
```

이름 붙은 단계에 대한 허가는 작업 수락이 아닙니다.

## 6. 근거와 확인 보여주기

구현 뒤 에이전트는 주장과 지원을 연결해야 합니다.

```text
주장:
로그인 유지 세션은 브라우저를 다시 열어도 유지됩니다.

근거:
- 변경된 로그인 폼과 세션 경로;
- 로그인 유지 세션과 아닌 세션에 대한 집중 테스트 출력;
- 세션 정리 경로를 살핀 메모.

확인:
- 집중 자동 테스트가 통과했습니다;
- 로그인 화면의 문구, 키보드 동작, 오류 상태 레이아웃은 아직 사람이 확인해야 합니다.
```

근거, 검증, 수동 QA는 관련되어 있지만 서로 다릅니다. 테스트 통과는 사람 경험을 증명하지 않습니다. 자체 확인은 충분히 분리된 검증보다 약합니다.

## 7. 수락 전에 잔여 위험 보여주기

사용자에게 결과 수락을 요청하기 전에 알려진 남은 불확실성을 보여줘야 합니다.

```text
잔여 위험:
로그인 유지 세션 동작은 로컬 브라우저 경로에서 확인했지만, 지원하는 모든 브라우저 정책 조합에서는 확인하지 않았습니다.
```

사용자가 이렇게 말해도:

```text
좋아 보여.
```

에이전트는 이 말을 이름 붙은 판단에 연결해야 합니다.

```text
이 작업을 닫으려면 확인이 필요합니다:
위 잔여 위험과 함께 로그인 유지 결과를 수락하시나요?
```

작업 수락은 사용자의 결과 판단입니다. 민감 단계 허가, 근거, 테스트, 수동 QA, 검증, 잔여 위험 표시와는 별개입니다.

## 8. 정직하게 닫기

좋은 닫기 요약은 짧고 근거 있어야 합니다.

```text
닫기 가능 여부:
- 범위는 로그인 체크박스, 선택한 세션 동작, 집중 테스트, 관련 문구 안에 머물렀습니다.
- 사용자 소유 제품 의미가 결정됐습니다.
- 근거는 로그인 유지 세션과 아닌 세션 동작을 뒷받침합니다.
- 수동 QA가 처리됐거나, 면제와 영향이 보입니다.
- 작업 수락 전에 잔여 위험을 보여줬습니다.
```

빠진 것이 있으면 가장 작은 해소 방법을 이름 붙입니다.

```text
닫기 막힘:
로그인 화면 수동 QA가 통과하지 않았고 명시적으로 면제되지도 않았습니다.
```

## 기억할 것

하네스는 작업 하나를 시작하기 어렵게 만드는 것이 아니라 믿기 쉽게 만드는 장치입니다. 사용자는 평소 말로 요청합니다. 에이전트는 필요할 때 구체화합니다. 사용자 소유 판단은 사용자에게 남습니다. 근거, 검증, 수동 QA, 작업 수락, 잔여 위험, 닫기 가능 여부는 서로 구분됩니다.
