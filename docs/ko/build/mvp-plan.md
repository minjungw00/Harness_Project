# Build: MVP 계획

서버 구현 전 문서 계획을 위한 Build 진입점입니다. 저장소 상태, 구현 준비 결정, 첫 스모크 목표의 의도, 문서 계획 종료 기준을 기록합니다.

Build 문서는 계획 안내만 담당합니다. 정확한 현재 MVP 범위, 스키마, enum 값 집합, DDL, API 요청/응답 형태, 저장소 테이블, 상태 보기 템플릿 본문, fixture 형식, 보안 보장 주장은 정의하지 않습니다. 그런 계약은 [참조 색인](../reference/README.md)이 연결하는 담당 문서에 둡니다.

<a id="문서-수락-상태"></a>
## 저장소 상태

이 저장소는 현재 문서 전용이며 문서 검토 상태입니다. 향후 하네스 서버를 위한 원천 자료일 뿐입니다. 하네스 서버 구현, Harness Runtime Home, Product Repository, 런타임 기록 저장소, 구현 완료 동작이 아닙니다.

유지보수자가 [서버 코딩 전 결정](#서버-코딩-전-결정)의 구현 차단 결정을 해결하거나, 수락하거나, 이름 붙은 범위 영향과 함께 명시적으로 미루기 전까지 서버 코딩은 시작하면 안 됩니다.

## 현재 있는 것

- Start, Use, Build, Reference, Later, Maintain 구조 안에 놓인 한영 문서 동시 유지 계획 문서.
- 현재 MVP 범위, Core, API, 저장소, 런타임 경계, 보안, 에이전트 통합, 상태 보기 권한, 템플릿 본문, 적합성, 설계 품질, 용어를 맡는 기준 참조 담당 문서.
- 현재 MVP 밖 자료를 모아 두는 단일 이후 후보 색인.
- 한영 검색 경로와 `doc_id`별 한 언어 맥락 규칙을 다루는 `docs/doc-index.yaml`.
- 구현 준비 진입점인 이 Build 계획.

## 현재 없는 것

- 서버/런타임 구현.
- 실행 가능한 적합성 실행기.
- 생성된 런타임 보고서.
- 런타임 상태나 생성된 상태 보기.
- OS 수준 권한 제어.
- 임의 도구 샌드박스.
- 변조 방지 저장소.
- 기본 도구 실행 전 차단.
- 활성 운영 프로필.

## 현재 MVP 범위

현재 MVP 범위의 기준 담당 문서는 [현재 MVP 범위](../reference/active-mvp-scope.md)입니다. Build는 범위 목록을 다시 정의하지 않습니다. 범위 문구가 바뀌면 먼저 범위 담당 문서를 고치고, 이 계획은 준비 상태와 순서 경로로 유지합니다.

## 제외되는 이후 자료

이후 후보와 승격 경계는 [이후 후보 색인](../later/index.md)이 담당합니다. Build는 준비 상태 영향을 설명할 때만 이후 제외 항목을 이름 붙일 수 있습니다. 이후 후보 계약을 정의하거나 현재 MVP로 승격하지 않습니다.

<a id="첫-내부-스모크-목표"></a>
## 첫 내부 스모크 목표

첫 내부 스모크 목표는 문서 스모크 목표입니다. 제품 MVP도 아니고, 완전한 적합성 테스트 모음도 아니며, 구현 계획도 아닙니다.

이 목표는 향후 첫 런타임 작업 묶음에서 위험한 담당 문서 경계를 건드려야 합니다. 현재 MVP 범위, Core 전이, API 응답 분기, 저장 효과, 아티팩트 스테이징과 승격, 사용자 소유 판단, 닫기 준비 상태, 보안 보장 표현, 커넥터 대체 동작, 파생 표시 권한을 피하지 않는 것이 목적입니다.

정확한 스모크 예시, fixture 형태, 공개 오류, 스키마, 상태 효과, 저장 결과는 기준 담당 문서에서 가져와야 합니다. 이 계획은 스모크 목표가 왜 필요한지만 기록합니다. 실행 가능한 fixture, 생성된 보고서, 런타임 상태, 구현 허가를 만들지 않습니다.

## 사용자 작업 루프

사용자 작업 루프는 사용자가 하네스 내부 라벨을 몰라도 평소 작업을 시작하거나 이어갈 수 있게 해야 합니다. 사용자가 원하는 것, 저장소나 향후 하네스 상태가 뒷받침할 수 있는 것, 아직 불확실한 것, 사용자가 직접 판단해야 하는 것을 구체화해야 합니다.

다음 안전한 행동은 계속 보여야 합니다. Core, MCP, 기준 접점이 어떤 주장을 뒷받침할 수 없다면 상태가 그 사실을 말해야 합니다. 권한을 지어내면 안 됩니다.

## 요청에서 닫기까지의 계획 경로

요청에서 닫기까지의 각 계약은 [참조 색인](../reference/README.md)에서 정확한 담당 문서를 골라 확인합니다. Build는 계획 순서만 남깁니다.

1. 평소 말에서 Task를 만들거나 이어갑니다.
2. 범위와 사용자 소유 판단은 담당 문서가 정한 경로로 기록합니다.
3. 제품 쓰기 전에는 제품 쓰기 호환성을 확인합니다.
4. 실행과 오래 남는 증거 참조는 담당 문서가 정한 경로로 기록합니다.
5. 상태와 간결한 출력은 파생 읽기로 보여 줍니다.
6. 닫기 준비 상태를 확인하되 증거, 최종 수락, 잔여 위험 수락, later QA/검증 후보를 서로 구분합니다.

`compatible`, `blocked`, `allowed`는 향후 하네스 기록 호환성 결과입니다. 향후 승격된 메커니즘이 정확한 동작을 증명하지 않는 한 물리적 OS 차단, 임의 도구 실행 방지, 샌드박스 격리, 권한 격리를 뜻하지 않습니다.

<a id="서버-코딩-전-결정"></a>
## 서버 코딩 전 결정

유지보수자가 각 행을 수락하거나, 결정하거나, 명시적인 범위 영향과 함께 미루기 전까지 서버 코딩은 시작하면 안 됩니다.

| 결정 항목 | 현재 상태 | 코딩 전에 결정할 것 |
|---|---|---|
| 구현 계획 준비 판단 | 유지보수자 수락 전입니다. | 현재 간결한 문서 묶음이 첫 런타임 작업 묶음 계획에 충분한지 유지보수자가 수락하거나, 차단 사유와 영향을 받는 범위를 이름 붙여야 합니다. |
| 현재 MVP 범위 수락 | 코딩용 유지보수자 수락 전입니다. | [현재 MVP 범위](../reference/active-mvp-scope.md) 경계를 유지보수자가 수락하거나 해결되지 않은 범위 영향을 이름 붙여야 합니다. |
| Core와 사용자 판단 수락 | 코딩용 유지보수자 수락 전입니다. | Core 전이 의미와 사용자 소유 판단 경계가 현재 MVP 경로 기준으로 유지보수자에게 수락되어야 합니다. |
| 공개 API와 스키마 수락 | 코딩용 유지보수자 수락 전입니다. | 메서드 동작, 공통 envelope/응답 분기, 상태/아티팩트/판단 스키마, 값 집합, 공개 오류, 멱등성/replay 동작, Core/MCP를 사용할 수 없을 때의 동작이 영향을 받는 도구나 리소스 코딩 전에 수락되어야 합니다. |
| 저장소와 런타임 홈 수락 | 코딩용 유지보수자 수락 전입니다. | DDL, 런타임 데이터 파일, 아티팩트 저장소를 만들기 전에 저장소 기록, 저장 효과, 아티팩트 생명주기, 버전 관리, 잠금, 마이그레이션, Runtime Home 경계가 수락되어야 합니다. |
| 보안과 로컬 접근 수락 | 코딩용 유지보수자 수락 전입니다. | API/MCP 노출 전에 로컬 전용 태세, 협력형/제한된 탐지형 보장 표현, 비보장이 수락되어야 합니다. |
| 접점과 간결한 출력 수락 | 코딩용 유지보수자 수락 전입니다. | 표시 코드나 커넥터 코드를 구현하기 전에 커넥터 동작, 접점별 사용법, 간결한 표시 경계, 최신성/사용 불가 동작, 상태 보기 권한, 템플릿 본문이 수락되어야 합니다. |

## 담당 참조 문서

기준 담당 문서 지도는 [참조 색인](../reference/README.md)을 사용합니다. Build는 색인이 경로 기준이므로 담당 문서 표를 중복하지 않습니다.

## 문서 계획 종료 기준

문서 계획은 유지보수자가 아래 항목을 명시적으로 확인한 뒤에만 종료할 수 있습니다.

- 이 Build 계획이 활성 Build 진입점입니다.
- 위 구현 준비 결정마다 유지보수자의 결정, 수락, 또는 이름 붙은 범위 영향이 있는 보류가 있습니다.
- 현재 MVP 범위 담당 문서가 수락되었거나, 남은 경계 문제가 이름 붙은 범위 영향을 갖습니다.
- 현재 MVP에 필요한 Core, API, 저장소, 런타임 경계, 보안, 에이전트 통합, 상태 보기, 템플릿, 적합성, 설계 품질, 용어, 이후 후보 경계가 담당 참조 문서 사이에서 일치합니다.
- 영어와 한국어 Build 문서가 같은 구현 결정을 보존합니다.
- 이후 후보 자료가 현재 MVP 필수 요구사항처럼 제시되지 않습니다.
- 문서는 원천 자료로만 남아 있으며 이곳에 서버/런타임 코드, 생성된 런타임 상태, 실행 가능한 fixture, 적합성 결과, 생성된 런타임 보고서, 제품 구현 출력물이 만들어지지 않았습니다.

이 문서 계획 기준을 통과해도 하네스가 구현되거나, 런타임 적합성이 증명되거나, 향후 제품 작업이 닫히는 것은 아닙니다.
