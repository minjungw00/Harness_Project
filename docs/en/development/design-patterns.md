# Implementation design patterns

This guide explains stable implementation structures visible in the current
Rust workspace. It is a developer-learning document: it names problems,
source locations, benefits, trade-offs, and tests, but it does not define
public API behavior, schemas, storage effects, security guarantees, runtime
behavior, or Core authority semantics.

Use wording such as "resembles" or "functions as" here deliberately. The
source shows durable structures, but this page does not claim formal pattern
intent beyond what the code demonstrates.

For exact product behavior, use the [Reference Index](../reference/README.md).
For the broad crate map, use [Implementation Architecture](architecture.md);
for representative method traces, use [Request Lifecycle](request-lifecycle.md).

## Pattern Map

| Structure | Problem addressed | Source locations and symbols | Benefit | Trade-off | Tests |
|---|---|---|---|---|---|
| Ports-and-adapters-like dependency direction | Public method behavior needs to stay independent of transport and setup surfaces. | [`crates/volicord-core/src/pipeline.rs`](../../../crates/volicord-core/src/pipeline.rs) for `CoreService`; [`crates/volicord-mcp/src/lib.rs`](../../../crates/volicord-mcp/src/lib.rs) for `McpAdapter`; Cargo manifests for `volicord-core`, `volicord-mcp`, and `volicord-cli`. | Core can be exercised directly and through MCP without adapter-owned method semantics. | Adapter startup and session validation still need Store access, so readers must separate startup validation from public method execution. | `adapter_and_direct_core_status_have_equivalent_response_meaning` in [`crates/volicord-mcp/src/lib.rs`](../../../crates/volicord-mcp/src/lib.rs); `mcp_session_derives_access_per_method_call` in [`tests/integration/mcp_surface.rs`](../../../tests/integration/mcp_surface.rs). |
| Application-service-like `CoreService` | Public methods need a single adapter-independent entry point over Runtime Home, ID generation, time, Store access, and response construction. | `CoreService`, `CoreService::prepare_request`, and `CoreService::execute_prepared_request` in [`crates/volicord-core/src/pipeline.rs`](../../../crates/volicord-core/src/pipeline.rs); method entry functions in [`crates/volicord-core/src/methods/`](../../../crates/volicord-core/src/methods/). | Common preflight and response rules are centralized while method files keep method-specific planning. | `CoreService` becomes a coordination point; changes to common preflight can affect every method branch. | Pipeline tests such as `rejected_branch_has_no_storage_effect`, `committed_mutation_increments_state_version_once`, and `access_class_mismatch_is_rejected_without_effect` in [`crates/volicord-core/src/pipeline.rs`](../../../crates/volicord-core/src/pipeline.rs). |
| Reusable policy object | Method preflight needs consistent access, Task, replay, freshness, and effect requirements without duplicating the sequence in each method. | `MethodPolicy`, `TaskRequirement`, `ReplayPolicy`, `FreshnessPolicy`, and `MethodEffectPolicy` in [`crates/volicord-core/src/pipeline.rs`](../../../crates/volicord-core/src/pipeline.rs); `mutation_method_policy` in [`crates/volicord-core/src/methods/mod.rs`](../../../crates/volicord-core/src/methods/mod.rs). | Method code declares the required policy while the shared pipeline owns the ordering. | A policy is only the common gate. Method-specific compatibility still lives in planners and Reference owners. | `stale_expected_state_version_is_rejected_without_effect` and `idempotency_replay_returns_stored_response` in [`crates/volicord-core/src/pipeline.rs`](../../../crates/volicord-core/src/pipeline.rs). |
| Planning before effect execution | Methods need to decide validation, result fields, events, and storage mutations before any normal Core mutation is committed. | `plan_intake` in [`crates/volicord-core/src/methods/intake.rs`](../../../crates/volicord-core/src/methods/intake.rs), `plan_prepare_write` in [`crates/volicord-core/src/methods/prepare_write.rs`](../../../crates/volicord-core/src/methods/prepare_write.rs), `OwnerPipelineBranch` in [`crates/volicord-core/src/pipeline.rs`](../../../crates/volicord-core/src/pipeline.rs). | Dry-run, rejected, no-effect, and committed branches are visibly separated. | Planners must compute projected response facts before Store commit facts are final, so commit-time facts are passed back through `CommittedMutationFacts`. | `intake_dry_run_has_no_storage_effect`, `prepare_write_dry_run_has_no_authorization_effect`, and `prepare_write_allowed_creates_one_authorization_with_post_commit_basis` in [`crates/volicord-core/src/methods/tests.rs`](../../../crates/volicord-core/src/methods/tests.rs). |
| Command-like storage mutation values | Method planners need a typed way to tell Store what to apply without embedding SQL or storage ordering in method code. | `CoreStorageMutation` and `ProjectMutation` in [`crates/volicord-store/src/core_pipeline.rs`](../../../crates/volicord-store/src/core_pipeline.rs); mutation lists built in method planners such as `intake.rs`, `prepare_write.rs`, and `record_run.rs`. | Store owns persistence mechanics while Core planners own method intent. | Adding a new persistent effect requires updating both planner output and Store mutation application, plus Reference owners for exact behavior. | `task_and_judgment_basis_store_apis_round_trip` in [`crates/volicord-store/src/core_pipeline.rs`](../../../crates/volicord-store/src/core_pipeline.rs); method effect tests in [`crates/volicord-core/src/methods/tests.rs`](../../../crates/volicord-core/src/methods/tests.rs). |
| Explicit transaction and atomic commit boundary | Normal committed method effects need replay, freshness, state-version change, events, response JSON, and storage mutations to succeed or fail together. | `CommitMutationInput`, `CommittedMutationFacts`, `MutationCommitOutcome`, and `CoreProjectStore::commit_mutation` in [`crates/volicord-store/src/core_pipeline.rs`](../../../crates/volicord-store/src/core_pipeline.rs); `begin_immediate_transaction` in [`crates/volicord-store/src/sqlite.rs`](../../../crates/volicord-store/src/sqlite.rs). | Commit behavior has one Store boundary instead of many partial writes. | Non-commit effect paths, especially transient artifact staging, must be documented and tested separately. | Store transaction tests such as `transaction_replay_returns_stored_response_before_stale_expected_state` and `transaction_replay_hash_conflict_rejects_without_effect`; pipeline test `committed_mutation_increments_state_version_once`. |
| Injectable clock | Authority-sensitive timestamps need deterministic tests without replacing production time behavior. | `Clock`, `SystemClock`, `CoreService::with_clock`, and `CoreService::now` in [`crates/volicord-core/src/pipeline.rs`](../../../crates/volicord-core/src/pipeline.rs); `ManualClock` in [`crates/volicord-core/src/methods/tests.rs`](../../../crates/volicord-core/src/methods/tests.rs); `FixedClock` in [`tests/conformance/baseline.rs`](../../../tests/conformance/baseline.rs). | Expiry, staging, and time-bound behavior can be exercised predictably. | Code that needs time must receive it through Core rather than calling the system clock directly. | Time-sensitive method tests in [`crates/volicord-core/src/methods/tests.rs`](../../../crates/volicord-core/src/methods/tests.rs) and conformance scenarios using `CoreService::with_clock`. |
| Opaque identifier newtypes | IDs need type separation while staying serializable as strings. | `opaque_string_type!` and wrappers such as `ProjectId`, `TaskId`, `RequestHash`, and `WriteAuthorizationId` in [`crates/volicord-types/src/ids.rs`](../../../crates/volicord-types/src/ids.rs). | Rust catches many accidental ID-family mixups, and serde still exposes the documented string-shaped API surface. | The wrapper does not itself validate product meaning; validation and authority stay with owners and implementation checks. | `sequence_generator_preserves_kind_prefixes` in [`crates/volicord-types/src/ids.rs`](../../../crates/volicord-types/src/ids.rs) and request-shape tests in [`crates/volicord-types/src/lib.rs`](../../../crates/volicord-types/src/lib.rs). |
| Enum-based controlled values | Public value sets and storage-facing controlled names need typed handling in Rust. | `MethodName`, `AccessClass`, `ResponseKind`, `EffectKind`, and related enums in [`crates/volicord-types/src/values.rs`](../../../crates/volicord-types/src/values.rs). | Adapters, Core, Store, and tests share one Rust representation for controlled values. | Exact value meaning still belongs to API value-set and method Reference owners; enum presence alone is not a product contract. | `typed_requests_derive_documented_access_classes` and closed-shape request tests in [`crates/volicord-types/src/lib.rs`](../../../crates/volicord-types/src/lib.rs). |
| Canonical request hashing | Idempotency replay needs request identity that is stable across JSON property ordering and formatting. | `canonical_json_bytes`, `canonical_json_sha256`, and `canonical_request_hash` in [`crates/volicord-types/src/canonical.rs`](../../../crates/volicord-types/src/canonical.rs); preflight hashing in [`crates/volicord-core/src/pipeline.rs`](../../../crates/volicord-core/src/pipeline.rs). | Replayed committed calls can compare a deterministic request hash instead of raw input text. | The hash depends on the typed request representation; schema changes must be considered with idempotency and Reference owners. | `canonical_json_hash_is_order_stable`, `typed_request_hash_ignores_raw_order_and_preserves_semantic_differences`, and `typed_request_hashes_are_stable_across_public_request_serialization` in [`crates/volicord-types/src/lib.rs`](../../../crates/volicord-types/src/lib.rs); `replay_hash_uses_decoded_typed_request_not_raw_property_order` in [`crates/volicord-mcp/src/lib.rs`](../../../crates/volicord-mcp/src/lib.rs). |
| Test fixture composition through `volicord-test-support` | Cross-layer tests need disposable Runtime Home setup, registered projects and surfaces, and request builders without putting fixture code in production crates. | `TempRuntimeHome`, `CoreFixture`, request builders, and Store inspection helpers in [`crates/volicord-test-support/src/lib.rs`](../../../crates/volicord-test-support/src/lib.rs). | Tests can compose Store, Core, MCP, and conformance paths consistently while keeping runtime data under temporary directories. | Fixture helpers can make setup easy to reuse; assertions still need to route back to Reference owners for product meaning. | `tests/conformance/baseline.rs`, `tests/integration/mcp_surface.rs`, and helper tests such as `disposable_runtime_home_stays_under_system_temp` in [`crates/volicord-test-support/src/lib.rs`](../../../crates/volicord-test-support/src/lib.rs). |

## How To Use This Page

When making a narrow change, use this page to identify the implementation
structure you are touching. Then use the [Implementation Guide](change-guide.md)
to find the exact Reference owner and test layer. Do not add new behavior
because it "fits" a pattern; add behavior only when the applicable owner defines
it or when an owner gap has been resolved.
